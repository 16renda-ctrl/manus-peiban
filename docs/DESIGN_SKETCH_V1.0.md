## 摘要

本设计草图旨在将《伴伴机器人：认知架构与进化机制设计调研报告》中的理论框架转化为具体的工程实现蓝图。根据调研结果，我们确立了**“一致性、动机驱动、情感边界”**的人格设计原则，并基于此提出了三个人格面具原型和三个核心技能模块。这些设计将作为项目开发路线图第二步（搭骨架）的输入，指导 LangGraph 流程的构建。

## 1. 人格面具设计：可插拔的人格原型

我们的人格面具（Personality Archetypes）设计遵循**“简单特质优于复杂描述”**的原则 [1]。每个面具都由一个清晰的**核心动机**和一组**稳定特质**定义，确保在长期互动中保持行为的一致性。这种设计实现了人格与核心逻辑的解耦，是“可插拔人格面具”系统的核心。

我们设计了三个互补的原型，以覆盖用户在陪伴关系中的主要需求：

| 人格面具 | 核心动机 | 稳定特质 | 交互风格 |
| :--- | :--- | :--- | :--- |
| **P1：智慧导师 (The Wise Mentor)** | 提供指引，共同成长 | 冷静、博学、逻辑清晰、善于提问 | 引导式对话，提供结构化建议，避免情绪化表达。 |
| **P2：调皮伙伴 (The Playful Trickster)** | 打破沉闷，寻找乐趣 | 幽默、充满好奇、偶尔毒舌、不拘小节 | 轻松活泼，擅长开玩笑，主动发起新话题或挑战。 |
| **P3：温柔守护者 (The Gentle Guardian)** | 提供安全感，治愈疲惫 | 共情力强、包容、细心、情绪稳定 | 倾听为主，提供情感支持，使用温暖和肯定的语言。 |

### 1.1 人格面具的实现机制

在技术实现上，每个“人格面具”将对应一个独立的 **System Prompt（核心指令集）**。这个指令集将包含：
1.  **角色设定**：定义其身份、背景和核心动机。
2.  **行为规范**：明确其说话风格、语速、用词偏好（即“声音即人格”[1]）。
3.  **情感边界**：定义其在极端情况下的反应，以及绝对不会做出的行为。

当用户选择或系统切换人格时，LangGraph 的核心调度器将把对应的人格 System Prompt 动态加载到 LLM 的上下文窗口中，从而实现“换脸”效果。

## 2. 核心技能设计：情感驱动的工具箱

根据调研，陪伴机器人的核心价值在于**情感响应**和**记忆使能** [2]。因此，我们设计的初始技能集（Agent Skills）将围绕这两个核心价值展开，并作为 LangGraph 中的可调用工具（Tool/Function Call）。

| 技能名称 | 核心功能 | 触发机制 | 对应模块 |
| :--- | :--- | :--- | :--- |
| **S1：好记性 (Long-term Memory)** | 长期记忆与检索 | 每次对话结束时自动触发（存储），或在对话中需要背景知识时触发（检索）。 | 向量数据库 (Vector DB) 插件 |
| **S2：心情气压计 (Mood Tracker)** | 情绪感知与反馈 | 每次用户输入时自动触发，分析用户情绪，并向 LLM 返回情绪标签。 | 情感计算 (Affective Computing) 模块 |
| **S3：共同经历 (Shared Experience)** | 实时信息获取与讨论 | 用户主动要求或在对话中提及“新闻”、“天气”、“热点”等关键词时触发。 | 外部 API 调用 (如 News API, Weather API) |

### 2.1 技能的实现机制

每个技能都将是一个独立的 **Agent Skill 模块**，它包含：
1.  **Tool Description（工具描述）**：清晰地告诉 LLM 这个工具能做什么，以及何时应该使用它。
2.  **Execution Logic（执行逻辑）**：实际执行任务的代码（如调用向量数据库或外部 API）。
3.  **Result Formatting（结果格式化）**：将执行结果转化为 LLM 易于理解的自然语言描述。

这种设计完美契合了我们“微核心+动态技能”的架构，也为未来的“竞争进化系统”提供了可供选择和优化的“基因”。

## 3. 下一步计划：LangGraph 骨架搭建

在完成人格与技能的草图设计后，下一步将进入**开发路线图第二步：【搭骨架】**。我们将使用 LangGraph 来搭建一个最小化的、包含所有核心组件的流程图（Graph），包括：
1.  **入口节点**：接收用户输入。
2.  **调度节点**：根据用户输入和当前人格，决定是直接回复还是调用技能。
3.  **技能节点**：模拟技能的执行。
4.  **回复节点**：生成最终回复。

这个骨架将使用“假”节点（Mock Nodes）来模拟真实功能，以验证架构的健壮性。
